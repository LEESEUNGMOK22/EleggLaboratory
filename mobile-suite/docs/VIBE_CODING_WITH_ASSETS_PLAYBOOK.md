# Vibe Coding with Assets Playbook

공용 리소스를 활용해 여러 Android/Flutter 앱을 빠르게 만드는 실전 공략.

## 핵심 원칙

1. **원본과 실사용본을 분리**한다.
2. **작은 단위로 기능+리소스**를 함께 검증한다.
3. **앱 번들에는 필요한 파일만** 넣는다.
4. 생성 속도보다 **재현 가능한 구조**를 우선한다.

---

## 1) 스타트 템플릿 (권장)

- 공용 저장소: `mobile-suite/common/`
- 앱별 사용본: `mobile-suite/apps/<app>/assets/`

예시:

```text
apps/app-one/assets/
  ui/
  illustrations/
  audio/
  fonts/
```

공용에서 전부 직접 참조하기보다, 앱별로 사용하는 파일만 복사/링크하여 관리하면 릴리즈 크기 관리가 쉬움.

---

## 2) 바이브 코딩 루프 (리소스 포함)

### Step A. 프롬프트로 기능 단위 생성
예: “온보딩 3페이지 + 배경 일러스트 + 다음 버튼 애니메이션”

### Step B. 리소스 매핑표 작성
- 화면 ID
- 필요한 아이콘/이미지/폰트/사운드
- 파일 경로

### Step C. 즉시 실행 검증
- 화면 렌더
- 누락 에셋 에러 여부
- 로딩 지연 여부

### Step D. 정리 커밋
- 기능 코드 + 사용한 에셋 목록 + 테스트 로그

---

## 3) 리소스 선택 전략

### UI 패키지(SVG 대량) 사용시
- 먼저 앱의 핵심 20~50개만 채택
- 나머지는 후보군으로 보관

### 오디오 팩 사용시
- 알림/효과음 우선순위 매트릭스 작성
- 같은 의미의 음성은 1종만 기본값으로 채택

### 폰트 사용시
- 브랜드 타이틀/본문 분리
- 성능과 가독성 먼저 확인

---

## 4) AI에게 이렇게 요청하면 품질이 올라감

### 좋은 요청 예시
- “app-one의 홈 화면에 `ui/button_primary.svg`를 적용하고, 실패 시 fallback 버튼 스타일도 같이 구현해줘.”
- “`assets/audio/`에서 클릭 효과음 1개만 사용하도록 바꾸고 preload는 제거해줘.”
- “에셋 로딩 실패 시 기본 placeholder 이미지 보여주는 위젯 추가해줘.”

### 피해야 할 요청
- “공용 폴더 전부 앱에 연결해줘.”
- “그냥 예쁘게 다 넣어줘.”

(결과: 번들 비대화 + 유지보수 난이도 급상승)

---

## 5) 성능/용량 공략

- 이미지: PNG 고집보다 WebP/벡터 우선
- 에셋: 초기화면에 필요한 최소 세트만 선로딩
- 빌드 점검: `flutter build appbundle --analyze-size`
- 큰 팩은 온디맨드 성격으로 기능별 분할
